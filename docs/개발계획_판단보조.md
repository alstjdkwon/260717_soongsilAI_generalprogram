# 승인/반려 판단 보조 — 개발 계획

> 기준 문서: 개발계획.md (Phase 1~7) · 작성일: 2026-07-22
> 전제: Phase 1~6 완료, Phase 7(데모 마감) 진행 중

## 0. 목표와 범위

전세용 님이 매번 직접 작성하는 **승인/반려 사유를 데이터로 축적**하고, 그 축적분을 판례처럼 검색해 **새 신청 건의 판단 초안을 AI가 제시**한다. 최종 결정은 여전히 사람이 한다.

기존 Phase 5(직무 부합 근거 생성)와의 차이:

| | Phase 5 근거 생성 | 본 계획 판단 보조 |
|---|---|---|
| 출력 | 담당업무↔교육명 부합 근거문 | 승인/반려 **추천 + 사유 초안** |
| 참조 데이터 | 최근 교정 N건 (최신순) | 과거 **결정 이력** (유사순) |
| 축적 주체 | 근거문 교정 | 결정 + 사유 그 자체 |

## 1. 핵심 설계 결정

### D1. 초기에는 검색하지 않는다 — 전량 주입

결정 건이 50건 이하면 교육명·담당업무·결정·사유를 **전부 프롬프트에 넣는다**. 건당 2~3줄이라 컨텍스트에 여유 있게 들어가고, 검색 레이어를 끼우면 "관련 판례를 놓치는" 실패 지점만 새로 생긴다.

### D2. 검색이 필요해지면 임베딩 — 바이그램만으로는 부족

기존 `educationSimilarity`(문자 바이그램 Dice)는 Phase 4에서 잘 작동했지만, 그건 *같은 교육의 표기 차이*를 흡수하는 문제였다. 판례 검색은 *다른 교육인데 의미가 비슷한 것*을 찾아야 하므로 문자 겹침으로는 무너진다.

```
"파이썬 데이터분석 실무" vs "Python 머신러닝 기초"
→ 바이그램 겹침 ≈ 0.05. 의미상으론 명백히 유사한 판례.
```

`text-embedding-3-small` 코사인 + 기존 바이그램을 **보완적으로** 쓴다.

```
eduSim = max(코사인(교육명), educationSimilarity(교육명))
jobSim = 코사인(담당업무)
score  = 0.6 × eduSim + 0.4 × jobSim
```

바이그램은 "같은 교육 재신청"을 1.0으로 확실히 잡고, 임베딩은 글자가 달라도 의미가 가까운 걸 잡는다. `max`면 어느 쪽도 놓치지 않는다. **0.5 미만 컷, 상위 5건**만 주입 — 무관한 판례는 없느니만 못하다.

### D3. 교육명과 담당업무는 따로 잰다

두 축을 하나의 문자열로 합쳐 임베딩하면 "교육은 거의 동일한데 직무가 무관한 건"과 "둘 다 어중간한 건"이 같은 점수로 뭉개진다. 별도로 재서 가중합한다.

### D4. 벡터 DB는 쓰지 않는다

연간 수백 건 규모다. 1,000건 × 1536차원 전체 스캔 코사인은 JS로 수 밀리초. 임베딩은 JSON 배열로 컬럼에 넣고 전체 스캔한다. pgvector·인덱스 도입 이유 없음.

### D5. 사유 입력은 처음엔 자유 텍스트

실제 사유가 어떤 형태인지 모르는 상태에서 select 박스를 만들면 잘못된 선택지가 데이터를 왜곡한다. 1단계는 **자유 텍스트 + 플레이스홀더 가이드**로 받고, 실데이터 20건을 본 뒤 구조화한다.

## 2. 승인/반려 사유가 담아야 할 정보

**이 절이 이 기능의 성패를 가른다.** 판례 검색은 과거 사유가 새 건으로 *일반화 가능할 때만* 작동한다. "부적합"이라고만 적힌 100건보다 아래 요소를 갖춘 15건이 낫다.

### 2.1 필수 — 이게 없으면 판례로 못 쓴다

| # | 요소 | 왜 필요한가 | 나쁜 예 → 좋은 예 |
|---|---|---|---|
| R1 | **판단 축** | 어떤 차원에서 갈렸는지. 검색이 같은 축의 판례를 우선 매칭해야 한다 | "부적합" → "담당업무와 무관" |
| R2 | **결정적 사실** | 무엇과 무엇을 비교해 그 결론이 났는지. 이게 있어야 새 건이 유사한지 LLM이 판단한다 | "직무 무관" → "담당업무가 회계인데 교육은 UX 디자인이라 연결점 없음" |
| R3 | **자기완결성** | 판례는 **단독으로 검색돼 나온다**. 맥락 의존 표현은 검색 결과에서 의미를 잃는다 | "위와 동일", "지난번 얘기한 대로" → 문장 안에 근거를 다 담기 |

### 2.2 있으면 품질이 크게 오른다

| # | 요소 | 왜 필요한가 |
|---|---|---|
| R4 | **일반화 범위** | "어학은 원칙적으로 불인정, 단 해외업무 담당자는 예외" — 이 한 문장이 사례 10건보다 가치 있다. 규칙 자체가 명시되면 AI가 유사 건에 바로 적용한다 |
| R5 | **예외/조건부 표시** | "이번만 승인, 다음부터는 사전 협의 필요" — 이걸 안 적으면 AI가 일반 승인 판례로 학습해 잘못 확산시킨다 |
| R6 | **반례와의 구분** | "비슷한 A는 승인했지만 이 건은 B라서 반려" — 경계선을 직접 그어주는 정보. 가장 값지고 가장 안 적힌다 |

### 2.3 시스템이 자동으로 남겨야 하는 것 (사람이 쓸 필요 없음)

| # | 요소 | 왜 필요한가 |
|---|---|---|
| R7 | **결정 시점** | 정책은 바뀐다. 오래된 판례가 현재 기준과 충돌할 때 최신 쪽을 따르게 해야 한다. 유사도에 최신성 가중치를 섞기보다, **날짜를 프롬프트에 넣고 "최근 결정이 현재 정책 기준"이라고 명시**하는 편이 단순하고 잘 동작한다 |
| R8 | **담당업무 스냅샷** | ⚠️ `employees.job_role`은 **변한다**. 홍길동이 회계팀→인사팀으로 이동하면, 과거 "홍길동의 노무 교육 승인" 판례를 현재 job_role로 조인해 읽는 순간 의미가 뒤집힌다. **결정 시점의 담당업무를 판례 행에 복사 저장해야 한다** |

### 2.4 판단 축(R1)이 특히 중요한 이유 — LLM이 필요 없을 수도 있다

실제 반려 사유를 열어봤을 때 다수가 아래와 같다면, 이 기능의 설계 자체가 바뀐다.

```
잔여 포인트 부족 / 서류 미비 / 신청 기한 경과 / 이미 수강한 과정
```

이건 판단이 아니라 **규칙 체크**다. 조건문 몇 줄이면 되고, 오히려 LLM보다 정확하다. AI 판단 보조는 나머지 **진짜 경계 사례**에만 쓰면 된다.

→ 그래서 **1단계에서 사례 3~5건을 먼저 받아 이 질문(판단인가 규칙인가)에 답하는 것**을 최우선으로 둔다.

### 2.5 사유 입력 플레이스홀더 (초안)

구조화 대신 가이드 문구로 품질을 유도한다.

```
예: 담당업무가 회계인데 신청 교육은 UX 디자인이라 직무 연결점이 없어 반려.
    (무엇과 무엇을 비교해 그렇게 판단했는지, 예외 조건이 있다면 함께 적어주세요)
```

## 3. 단계별 계획

### 1단계 — 사례 소량 확보 + 설계 확정 (선행)

- 전세용 님께 **사례 3~5건** 요청 (반려 위주 + 고민했던 건 1~2건)
- 확인할 것: **판단 축의 분포**(규칙형 vs 판단형), 사유 문장의 길이·형식, 승인 시에도 기록을 남기는지
- **검증**: §2.4의 분기 결정 — 규칙형이 다수면 규칙 엔진 우선, 판단형이 다수면 계획대로 진행

### 2단계 — 결정 이력 저장 (스키마 + UI)

- `cases`에 승인 사유 필드 추가 — **현재는 `reject_reason`만 있어 반려 사유만 쌓인다.** 이대로 두면 판례가 전부 반려 쪽이라 AI가 반려로 편향된다
- 결정 시점 담당업무 스냅샷 저장 (R8)
- 승인/반려 액션에 사유 입력 + 플레이스홀더 가이드 (§2.5)
- **검증**: 승인·반려 각각 사유가 저장되고, 직원의 담당업무를 나중에 바꿔도 과거 판례의 담당업무가 그대로인지 단위 테스트

### 3단계 — 결정 이력 전량 주입 + 판단 초안 생성

- `src/ai/decision.ts` — 결정 이력을 프롬프트에 주입, 승인/반려 추천 + 사유 초안 + 신뢰도 생성 (D1: 검색 없이 전량)
- 상세 화면 심사 패널에 초안 표시 → 전세용 님이 수정·확정 → 수정본이 다시 판례로 축적
- **검증**: 초안이 §2.1의 R1·R2를 갖춘 문장으로 나오는지 (축 명시 + 비교 대상 명시)

### 4단계 — 실데이터 확보 + 홀드아웃 검증

- 2·3단계 화면을 보여드리며 **나머지 15~20건** 요청 (형식이 명확해 부담도 적고 품질도 좋다)
- **20건은 few-shot, 10건은 검증용 홀드아웃**으로 분리
- **검증**: `npm run verify:decision` — 홀드아웃 건에 대해 AI 추천이 전세용 님 실제 결정과 몇 건 일치하는지. 데모에서 "10건 중 8건 일치"는 근거문 샘플보다 설득력 있다

### 5단계 — (조건부) 임베딩 판례 검색

- **착수 조건**: 결정 건 50건 초과 **또는** 4단계 일치율이 낮고 원인이 "무관한 판례 주입"으로 확인될 때
- `cases.edu_embedding`, `employees.role_embedding` (JSON float[]) 추가
- 임베딩 생성 시점: 건이 **결정되는 순간**(판례가 되는 순간) 1회 + 조회할 새 건 1회
- **검증**: 전량 주입 대비 홀드아웃 일치율이 오르는지. 오르지 않으면 도입하지 않는다

## 4. 함정과 주의사항

| # | 함정 | 대응 |
|---|---|---|
| F1 | **반려 편향** — 승인은 사유를 안 남기고 반려만 필수라 판례가 한쪽으로 쏠린다 | 2단계에서 승인 사유 필드 추가. 최소한 "승인(사유 없음)"도 판례에 포함해 균형 |
| F2 | **담당업무 변경** — 현재 job_role로 조인하면 과거 판례 의미가 뒤집힌다 | 결정 시점 스냅샷 저장 (R8) |
| F3 | **정책 변화** — 오래된 판례가 현재 기준과 충돌 | 날짜를 프롬프트에 노출 + "최근 결정이 현재 정책 기준" 명시 (R7) |
| F4 | **무관한 판례 주입** — 검색 도입 후 점수 낮은 건까지 넣으면 품질이 떨어진다 | 0.5 미만 컷, 상위 5건 제한 (D2) |
| F5 | **가상 데이터로 프롬프트 튜닝** — Phase 3의 FakeParser 방식이 여기선 통하지 않는다. 양식은 구조가 예측 가능했지만, 심사 기준은 **내용 자체가 미지수이고 그게 바로 만들려는 것**이다 | 1단계 실사례 확보 전까지 프롬프트 튜닝하지 않는다 |
| F6 | **초기 세트 완벽주의** | 콜드스타트만 넘기면 된다. 30건 받느라 2주 지연되는 것보다 15건으로 시작해 운영하며 쌓는 편이 낫다 |

| **벡터 DB (pgvector 등)** | 규모가 수백~수천 건이라 전체 스캔으로 충분 (D4) |
| **정보 대조 로직의 LLM화** | `flags.ts`의 교육명·금액·이름 불일치 비교는 결정적 규칙 로직이다. 현 상태가 정답 |

## 0. 목표와 범위

전세용 님이 매번 직접 작성하는 **승인/반려 사유를 데이터로 축적**하고, 그 축적분을 판례처럼 검색해 **새 신청 건의 판단 초안을 AI가 제시**한다. 최종 결정은 여전히 사람이 한다.

기존 Phase 5(직무 부합 근거 생성)와의 차이:

| | Phase 5 근거 생성 | 본 계획 판단 보조 |
|---|---|---|
| 출력 | 담당업무↔교육명 부합 근거문 | 승인/반려 **추천 + 사유 초안** |
| 참조 데이터 | 최근 교정 N건 (최신순) | 과거 **결정 이력** (유사순) |
| 축적 주체 | 근거문 교정 | 결정 + 사유 그 자체 |

## 1. 핵심 설계 결정

### D1. 초기에는 검색하지 않는다 — 전량 주입

결정 건이 50건 이하면 교육명·담당업무·결정·사유를 **전부 프롬프트에 넣는다**. 건당 2~3줄이라 컨텍스트에 여유 있게 들어가고, 검색 레이어를 끼우면 "관련 판례를 놓치는" 실패 지점만 새로 생긴다.

### D2. 검색이 필요해지면 임베딩 — 바이그램만으로는 부족

기존 `educationSimilarity`(문자 바이그램 Dice)는 Phase 4에서 잘 작동했지만, 그건 *같은 교육의 표기 차이*를 흡수하는 문제였다. 판례 검색은 *다른 교육인데 의미가 비슷한 것*을 찾아야 하므로 문자 겹침으로는 무너진다.

```
"파이썬 데이터분석 실무" vs "Python 머신러닝 기초"
→ 바이그램 겹침 ≈ 0.05. 의미상으론 명백히 유사한 판례.
```

`text-embedding-3-small` 코사인 + 기존 바이그램을 **보완적으로** 쓴다.

```
eduSim = max(코사인(교육명), educationSimilarity(교육명))
jobSim = 코사인(담당업무)
score  = 0.6 × eduSim + 0.4 × jobSim
```

바이그램은 "같은 교육 재신청"을 1.0으로 확실히 잡고, 임베딩은 글자가 달라도 의미가 가까운 걸 잡는다. `max`면 어느 쪽도 놓치지 않는다. **0.5 미만 컷, 상위 5건**만 주입 — 무관한 판례는 없느니만 못하다.

### D3. 교육명과 담당업무는 따로 잰다

두 축을 하나의 문자열로 합쳐 임베딩하면 "교육은 거의 동일한데 직무가 무관한 건"과 "둘 다 어중간한 건"이 같은 점수로 뭉개진다. 별도로 재서 가중합한다.

### D4. 벡터 DB는 쓰지 않는다

연간 수백 건 규모다. 1,000건 × 1536차원 전체 스캔 코사인은 JS로 수 밀리초. 임베딩은 JSON 배열로 컬럼에 넣고 전체 스캔한다. pgvector·인덱스 도입 이유 없음.

### D5. 사유 입력은 처음엔 자유 텍스트

실제 사유가 어떤 형태인지 모르는 상태에서 select 박스를 만들면 잘못된 선택지가 데이터를 왜곡한다. 1단계는 **자유 텍스트 + 플레이스홀더 가이드**로 받고, 실데이터 20건을 본 뒤 구조화한다.

## 2. 승인/반려 사유가 담아야 할 정보

**이 절이 이 기능의 성패를 가른다.** 판례 검색은 과거 사유가 새 건으로 *일반화 가능할 때만* 작동한다. "부적합"이라고만 적힌 100건보다 아래 요소를 갖춘 15건이 낫다.

### 2.1 필수 — 이게 없으면 판례로 못 쓴다

| # | 요소 | 왜 필요한가 | 나쁜 예 → 좋은 예 |
|---|---|---|---|
| R1 | **판단 축** | 어떤 차원에서 갈렸는지. 검색이 같은 축의 판례를 우선 매칭해야 한다 | "부적합" → "담당업무와 무관" |
| R2 | **결정적 사실** | 무엇과 무엇을 비교해 그 결론이 났는지. 이게 있어야 새 건이 유사한지 LLM이 판단한다 | "직무 무관" → "담당업무가 회계인데 교육은 UX 디자인이라 연결점 없음" |
| R3 | **자기완결성** | 판례는 **단독으로 검색돼 나온다**. 맥락 의존 표현은 검색 결과에서 의미를 잃는다 | "위와 동일", "지난번 얘기한 대로" → 문장 안에 근거를 다 담기 |

### 2.2 있으면 품질이 크게 오른다

| # | 요소 | 왜 필요한가 |
|---|---|---|
| R4 | **일반화 범위** | "어학은 원칙적으로 불인정, 단 해외업무 담당자는 예외" — 이 한 문장이 사례 10건보다 가치 있다. 규칙 자체가 명시되면 AI가 유사 건에 바로 적용한다 |
| R5 | **예외/조건부 표시** | "이번만 승인, 다음부터는 사전 협의 필요" — 이걸 안 적으면 AI가 일반 승인 판례로 학습해 잘못 확산시킨다 |
| R6 | **반례와의 구분** | "비슷한 A는 승인했지만 이 건은 B라서 반려" — 경계선을 직접 그어주는 정보. 가장 값지고 가장 안 적힌다 |

### 2.3 시스템이 자동으로 남겨야 하는 것 (사람이 쓸 필요 없음)

| # | 요소 | 왜 필요한가 |
|---|---|---|
| R7 | **결정 시점** | 정책은 바뀐다. 오래된 판례가 현재 기준과 충돌할 때 최신 쪽을 따르게 해야 한다. 유사도에 최신성 가중치를 섞기보다, **날짜를 프롬프트에 넣고 "최근 결정이 현재 정책 기준"이라고 명시**하는 편이 단순하고 잘 동작한다 |
| R8 | **담당업무 스냅샷** | ⚠️ `employees.job_role`은 **변한다**. 홍길동이 회계팀→인사팀으로 이동하면, 과거 "홍길동의 노무 교육 승인" 판례를 현재 job_role로 조인해 읽는 순간 의미가 뒤집힌다. **결정 시점의 담당업무를 판례 행에 복사 저장해야 한다** |

### 2.4 판단 축(R1)이 특히 중요한 이유 — LLM이 필요 없을 수도 있다

실제 반려 사유를 열어봤을 때 다수가 아래와 같다면, 이 기능의 설계 자체가 바뀐다.

```
잔여 포인트 부족 / 서류 미비 / 신청 기한 경과 / 이미 수강한 과정
```

이건 판단이 아니라 **규칙 체크**다. 조건문 몇 줄이면 되고, 오히려 LLM보다 정확하다. AI 판단 보조는 나머지 **진짜 경계 사례**에만 쓰면 된다.

→ 그래서 **1단계에서 사례 3~5건을 먼저 받아 이 질문(판단인가 규칙인가)에 답하는 것**을 최우선으로 둔다.

### 2.5 사유 입력 플레이스홀더 (초안)

구조화 대신 가이드 문구로 품질을 유도한다.

```
예: 담당업무가 회계인데 신청 교육은 UX 디자인이라 직무 연결점이 없어 반려.
    (무엇과 무엇을 비교해 그렇게 판단했는지, 예외 조건이 있다면 함께 적어주세요)
```

## 3. 단계별 계획

### 1단계 — 사례 소량 확보 + 설계 확정 (선행)

- 전세용 님께 **사례 3~5건** 요청 (반려 위주 + 고민했던 건 1~2건)
- 확인할 것: **판단 축의 분포**(규칙형 vs 판단형), 사유 문장의 길이·형식, 승인 시에도 기록을 남기는지
- **검증**: §2.4의 분기 결정 — 규칙형이 다수면 규칙 엔진 우선, 판단형이 다수면 계획대로 진행

### 2단계 — 결정 이력 저장 (스키마 + UI)

- `cases`에 승인 사유 필드 추가 — **현재는 `reject_reason`만 있어 반려 사유만 쌓인다.** 이대로 두면 판례가 전부 반려 쪽이라 AI가 반려로 편향된다
- 결정 시점 담당업무 스냅샷 저장 (R8)
- 승인/반려 액션에 사유 입력 + 플레이스홀더 가이드 (§2.5)
- **검증**: 승인·반려 각각 사유가 저장되고, 직원의 담당업무를 나중에 바꿔도 과거 판례의 담당업무가 그대로인지 단위 테스트

### 3단계 — 결정 이력 전량 주입 + 판단 초안 생성

- `src/ai/decision.ts` — 결정 이력을 프롬프트에 주입, 승인/반려 추천 + 사유 초안 + 신뢰도 생성 (D1: 검색 없이 전량)
- 상세 화면 심사 패널에 초안 표시 → 전세용 님이 수정·확정 → 수정본이 다시 판례로 축적
- **검증**: 초안이 §2.1의 R1·R2를 갖춘 문장으로 나오는지 (축 명시 + 비교 대상 명시)

### 4단계 — 실데이터 확보 + 홀드아웃 검증

- 2·3단계 화면을 보여드리며 **나머지 15~20건** 요청 (형식이 명확해 부담도 적고 품질도 좋다)
- **20건은 few-shot, 10건은 검증용 홀드아웃**으로 분리
- **검증**: `npm run verify:decision` — 홀드아웃 건에 대해 AI 추천이 전세용 님 실제 결정과 몇 건 일치하는지. 데모에서 "10건 중 8건 일치"는 근거문 샘플보다 설득력 있다

### 5단계 — (조건부) 임베딩 판례 검색

- **착수 조건**: 결정 건 50건 초과 **또는** 4단계 일치율이 낮고 원인이 "무관한 판례 주입"으로 확인될 때
- `cases.edu_embedding`, `employees.role_embedding` (JSON float[]) 추가
- 임베딩 생성 시점: 건이 **결정되는 순간**(판례가 되는 순간) 1회 + 조회할 새 건 1회
- **검증**: 전량 주입 대비 홀드아웃 일치율이 오르는지. 오르지 않으면 도입하지 않는다

## 4. 함정과 주의사항

| # | 함정 | 대응 |
|---|---|---|
| F1 | **반려 편향** — 승인은 사유를 안 남기고 반려만 필수라 판례가 한쪽으로 쏠린다 | 2단계에서 승인 사유 필드 추가. 최소한 "승인(사유 없음)"도 판례에 포함해 균형 |
| F2 | **담당업무 변경** — 현재 job_role로 조인하면 과거 판례 의미가 뒤집힌다 | 결정 시점 스냅샷 저장 (R8) |
| F3 | **정책 변화** — 오래된 판례가 현재 기준과 충돌 | 날짜를 프롬프트에 노출 + "최근 결정이 현재 정책 기준" 명시 (R7) |
| F4 | **무관한 판례 주입** — 검색 도입 후 점수 낮은 건까지 넣으면 품질이 떨어진다 | 0.5 미만 컷, 상위 5건 제한 (D2) |
| F5 | **가상 데이터로 프롬프트 튜닝** — Phase 3의 FakeParser 방식이 여기선 통하지 않는다. 양식은 구조가 예측 가능했지만, 심사 기준은 **내용 자체가 미지수이고 그게 바로 만들려는 것**이다 | 1단계 실사례 확보 전까지 프롬프트 튜닝하지 않는다 |
| F6 | **초기 세트 완벽주의** | 콜드스타트만 넘기면 된다. 30건 받느라 2주 지연되는 것보다 15건으로 시작해 운영하며 쌓는 편이 낫다 |

## 5. 채택하지 않은 선택지

| 선택지 | 판단 |
|---|---|
| **LangGraph** | 다단계 도구 호출 루프·체크포인트·재시도 분기가 필요할 때 가치가 있다. 현재 파이프라인은 "검색 → 생성 → 사람 확인"의 선형 3단계라 프레임워크가 해줄 일이 없고, 의존성과 실패 지점만 늘린다. 정책 문서가 여러 개 생겨 AI가 무엇을 조회할지 스스로 정해야 하는 시점에 재검토 |
| **Claude 리서치 스킬류** | Claude Code(개발 도구) 안에서 도는 것이라 Railway 배포 앱 런타임에 넣을 수 없다. 앱 내 에이전틱 루프가 필요해지면 OpenAI SDK tool-calling 또는 Claude API를 직접 쓴다 |
| **벡터 DB (pgvector 등)** | 규모가 수백~수천 건이라 전체 스캔으로 충분 (D4) |
| **정보 대조 로직의 LLM화** | `flags.ts`의 교육명·금액·이름 불일치 비교는 결정적 규칙 로직이다. 현 상태가 정답 |
