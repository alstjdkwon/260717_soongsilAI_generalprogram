# 숭실대 AX 공모전 — 클로드코드 인계 문서
> 작성일: 2026-07-23 · 이 문서는 별도 채팅 세션에서 진행된 기획 과정의 컨텍스트 인계용입니다.
> 프로젝트: 총무·인사팀 전세용 님(교육훈련·비정규직 인사 담당)의 자율교육 업무를 대상으로 한 AI 도구 개발

## 2. 공식 요강 핵심

**주제**: 숭실대학교의 교육·연구·행정 업무에 대해 생성형 AI를 활용한 개선 사례 제출

**제출물**: A4 5장 이내 보고서 + 증빙자료(형태 제한 없음)
- ⚠️ **"개선 전/후 비교는 반드시 포함", "정량/정성적 성과를 명확하게 작성"** — 필수 요건. 측정 설계 제시만으로는 부족하고 실측 데이터가 있어야 함.

**일정**: 보고서 접수(8월 말) → 1차 서류평가(9월 초) → 발표평가(9월 말) → 평가결과 확정(10월 초) → 결과공지·시상(개교기념식)

**참가 형태**: 개인/팀 모두 가능, 5인 이상은 인원별 역할 명시. **"직원-학생 혼성팀 가능"** — 이 조항이 중요함. 학생이 만들고 실제 직원이 사용하는 형태를 주최측이 애초에 전제하고 있다는 근거로, 인터뷰이에게 실사용을 요청할 때 "특혜 요청"이 아니라 "공모전이 원래 요구하는 형태"라는 논리로 쓸 수 있음.

**기타**: 교원 등 전문가 도움/타 대학 벤치마킹 가능하나 보고서에 명시 및 본교 적용 결과 포함 필요. 생성형 AI 등 개발자원 지원, 발표평가 참여자에게 구독비용 정액 지원.

**평가 방식**: 1차 서류평가(보고서+증빙) → 2차 발표평가(발표+질의응답). 1차 통과팀에 전문가 멘토링 지원. 평가위원 1·2차 각 5~6명.

**배점 (시상 기준: 각 요소 최소 60점 이상)**
| 평가분야 | 배점 | 핵심 세부요소 |
|---|---|---|
| 문제 정의의 적절성 | 10점 | 업무 문제 인식의 정확성, 개선 필요성 |
| AI 활용 적절성 | 20점 | AI 활용 필수 여부·적절성, 기존 방식 대비 차별성 |
| 실행 내용의 완성도 및 창의성 | 20점 | **실제 활용 과정에서의** AI 활용 완성도·창의성 |
| 업무 개선 정도 | 25점 | 시간·비용·정확성 등 정량적 개선 정도 (정량 어려우면 명확한 전/후 개선) |
| 확산 가능성 | 25점 | 타 부서·유사 업무 활용 가능성, 전 직원 전파 필요 우수사례 해당 여부 |

**핵심 시사점**: 배점 70점(실행완성도+업무개선+확산가능성)이 "실제 활용 여부"에 걸려 있음. 프로토타입만으로는 부족하고, **실사용자 확보 + 전/후 데이터 확보가 병목**이 됨. 이게 확인된 시점부터 계획의 우선순위가 "완성도 높은 프로토타입"에서 "실사용 확보 속도"로 전환됨.

## 3. 적용한 방법론

**전체 프레임: 컨설팅 케이스 인터뷰 4단계**
`Clarification(업무범위 러프 파악) → Structuring(Where축+Why축 설계) → Analysis(인터뷰로 데이터 채움) → Solution(설계·개발)`

**부서/업무 스크리닝 — 3축**
- 반복성: 같은 형태의 일이 자주 발생하는가
- 규칙성(비정형→정형): 판단 기준이 문서화된 규칙으로 존재하는가, 인풋이 비정형이고 아웃풋이 정형인가
- 접근성: 작업이 열린 도구(엑셀·문서·메일)에서 일어나는가, 아니면 SAP 등 폐쇄 기간계 안에서 끝나는가 (학생 프로토타입 실현 가능성 결정)

**보조 평가축(케이스 확정 후 적용)**: 담당자의 고통 인식 여부, 시즌성/마감압박, 성과 측정 가능성, 데이터 확보 가능성(공개 자료만으로 데모 가능한지), 확장성.

**Why축(원인 분석) — 시간방정식 4요인** (창업지원단 케이스에서 도출, 이후 범용 렌즈로 재사용)
```
행정 총시간 = Σ(건수 × 건당시간 × 재작업배수) + 횡단관리시간
```
→ 건수과다 / 건당시간과다 / 재작업과다 / 횡단오버헤드. 어느 부서·업무가 병목으로 확정되든 이 4칸에 데이터를 채우는 방식으로 분석.

**핵심 인터뷰 원칙**
1. **열린 질문 먼저, 가설은 체크리스트로 나중에.** 가설을 먼저 제시하면 인터뷰이가 그 틀 안에서만 답해 유도신문이 됨. 순서: (1)열린 질문으로 전체 프로세스 청취 → (2)"가장 시간 드는 단계는?"으로 자발적 병목 청취 → (3)언급 안 된 가설만 후반에 확인.
2. **AI는 판단이 아니라 근거 제시.** 재량·정성 판단(직무 연관성 점수, 인사 재량 등)은 항상 사람 몫. AI는 정보 대령 + 근거 문장 제시 + 신뢰도 표기까지만.
3. **데스크 리서치는 후보를 좁히는 데까지만 신뢰.** "진짜 아픈지"는 실제 담당자만 답할 수 있음. 이 원칙이 반복 검증됨 — 창업지원단(정산·보고 반복 가설이 매니저 피드백으로 전면 기각), 학사팀 김예은(예외처리 유형은 문서화돼 있었지만 실제 발생 건수는 적음), 전세용(법정교육은 본인 소관 아님) 케이스 모두 데스크 리서치 가설이 현장에서 뒤집힘.
4. **프로세스는 바꾸지 않고 상태추적 부담만 제거.** "프로세스 파괴"가 아니라 "프로세스는 그대로, 상태가 스스로 흐르게" — 권한도 없고, 형평성에 예민한 인터뷰이에게도 안전한 방향.
5. **MECE 점검 시**: Where축은 ME(상호배제)뿐 아니라 CE(전체포괄)도 확인 — 특정 단계에 안 속하는 횡단업무 버킷이 빠지기 쉬움. Why축은 위 시간방정식처럼 항등식으로 분해해야 MECE가 보장됨(임의로 나열한 가설은 MECE가 아님).

## 4. 부서 탐색 히스토리

| 부서/담당자 | 상태 | 판단 근거 (한 줄) |
|---|---|---|
| **창업지원단 (송수진 매니저)** | 인터뷰 완료, 사실상 종료 | 데스크리서치 가설("정산·보고 반복")이 매니저 피드백으로 전면 기각 → "수주(매번 창작이라 자동화 부적합)+운영" 재조사 → 이중입력은 확인됐으나 "그냥 보고 다시 치는" 방식이라 자동화 여지 낮음, 재작업은 비정형이라 패턴 없음, 공모 발굴·다중사업관리 가설도 기각됨 |
| **학사팀 김예은 (다전공·이수구분·졸업·제증명)** | 기각 | 데스크리서치로는 최유력(졸업사정 자동화 이미 존재, 예외처리 6종 taxonomy 확인)이었으나, 민성님의 실사용 지식으로 기각 — 이수구분 처리는 연 2회 10일만 접수, 단과대 조교의 선제 필터링으로 학사팀 도달 건수 자체가 적고 민원도 적음 |
| **장학팀** | 보류 (존재 확인 필요) | 초기 데스크리서치에서 RAG 적합도 최고로 평가됐으나, 최신 기구표 확인 결과 학생처 산하에 "장학팀" 명칭이 안 보임(학생지원팀에 흡수됐을 가능성) → 재확인 안 된 채 보류 |
| **총무·인사팀 전세용 (교육훈련·비정규직 인사)** | ★ 진행 중 (현재 케이스) | 인터뷰로 검증 완료 — 교육기획은 AX 부적합(본인 판단), 이수관리(그룹웨어→엑셀 수기입력)에서 휴먼에러+상태추적 부담(방치) 확인, SAP 미사용으로 접근성 통과. 자율교육 관리도구로 스코프 확정. 다만 사후 자료요청 이메일에 5일째 무응답 상태 |

**백업 후보 (5개 부서 데스크리서치 완료, 실제 인터뷰 미실시)**: 창업지원단·전세용 인터뷰에서 파생된 확장 후보로 **김정효 님(총무·인사팀, 복리후생·경조사·행사·서무)**이 유력 — 경조사 신고(비정형)→규정 대조→정형 처리 구조가 선명하고 SAP 밖일 가능성 높음, 전세용 님과 같은 팀이라 소개받기 용이. 그 외 중앙도서관·진로취업센터·상담센터는 데스크리서치만 완료된 상태.

## 7. 3종 교육(자율교육/인사팀주관교육/일반교육) 확장 구조 — 왜 이렇게 나눴는가

**출발점 — 세 유형은 업무 성격 자체가 다름**
- **자율교육**: 개인이 신청 → 심사 → 승인 → (교육 수강) → 이수 증빙 제출 → 환급. **신청 기반의 거래형 흐름**이라 상태기계(심사대기→이수대기→환급대기→완료)가 자연스럽게 맞음.
- **인사팀주관교육·일반교육**: 개인이 신청해서 승인받는 구조가 아니라, 여러 경로(내부 행사 출석부, 업체 결과물, 외부 이수증 등)에서 완료 증빙이 흘러들어오면 그걸 **개인별로 누적 집계**하는 구조. "승인/반려" 개념이 약한 대신, 세영 님이 인터뷰에서 명확히 선을 그은 **정성적 직무연관성 판단**(자율교육엔 없는 요소)이 존재.

**그래서 상태모델을 하나로 통일하면 안 됨** — "이수 대기" 같은 상태가 인사팀주관·일반교육엔 안 맞고, 억지로 맞추면 어색한 UX가 나옴. 대신 코어/모듈 구조로 분리:

- **코어 (3종 공유)**: 문서 업로드 드롭존, AI 추출+신뢰도 판정, 사람 검토 화면 패턴, **사람(직원) 중심의 기록 저장소** — 신청/이수 건이 "건" 중심이 아니라 "사람"에 딸리는 구조. 이래야 나중에 "이 직원이 지금까지 받은 교육 총합"이라는 최종 그림(승진 요건 판단용 — 세영 님이 실제로 엑셀로 180명분 관리 중이던 바로 그 화면)이 자연스럽게 나옴.
- **모듈 (업무별 로직 다름)**: 자율교육 모듈(신청-승인-환급 상태기계, 완전 구현 대상) / 인사팀주관·일반교육 모듈(여러 경로 증빙 → 개인별 누적 집계 + 정성판단 보조, 최소기능만).

**최근 변경사항 — 원래는 "확장 훅만, 지금 구현 안 함"이었으나 요강 확인 후 수정됨**
당초에는 자율교육만 완전 구현하고 나머지 두 종류는 데이터 필드·탭 자리 같은 확장 훅만 심어두는 계획이었음(코어/모듈 구조 자체는 그때부터 확정). 그런데 공식 요강의 **확산 가능성(25점) + 실행 완성도(20점)** 배점을 확인한 뒤, 자율교육 하나만으로는 부족하다고 판단 → 인사팀주관·일반교육도 **최소기능(여러 경로 이수 증빙 → 개인별 누적 집계)까지는 실제로 구현**하는 것으로 계획 수정. 다만 이쪽은 승인 플로우가 없어 자율교육보다 오히려 구조가 단순하다는 점을 활용해 개발 부담을 낮춤. 정성판단 로직(직무연관성 점수화)은 여전히 사람이 하되, AI가 상식 추론 기반 근거문 초안을 제시하고 사람 교정이 축적되는 구조(콜드스타트 대응)로 자율교육의 심사 단계 설계를 그대로 재사용.

**지금 코어에 반영해둔 확장 훅 3가지** (자율교육 개발 시점부터 미리 심어둔 것)
1. 데이터에 "교육유형" 필드 포함 (현재값은 "자율교육"뿐)
2. 기록 저장소를 사람(직원) 중심으로 설계 — 위 설명대로
3. 상단 탭 자리 — [자율교육](활성) / [인사팀주관교육] / [일반교육] (준비중 → 이제 이 두 개도 최소기능 구현 대상으로 전환)
