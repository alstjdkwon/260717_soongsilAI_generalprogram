## 0. 전제와 가정 (확인 필요 시 수정)

| # | 가정 | 근거 |
|---|---|---|
| A1 | **로컬 실행 웹앱** — 세영 님 PC에서 단일 프로세스로 실행, 서버 배포 없음 | 본인 전용 내부 도구, 그룹웨어 연동 스코프 밖 |
| A2 | **SQLite 파일 DB** — 브라우저 스토리지 대신 명시적 로컬 저장소 | 기획서 §8 "브라우저 스토리지 사용 불가 환경 가능성" |
| A3 | **OCR = Upstage / 추론 = OpenAI(GPT-5.6 Terra)** 로 역할 분리 | 사용자 결정. OCR은 Upstage, 문서 종류 판별·필드 구조화·대조·직무부합 추론은 OpenAI |
| A4 | 실제 신청서 양식 도착 전까지는 **가상 양식 PDF를 직접 만들어** 개발·테스트. 신청서 파서는 양식 도착 시 규칙기반으로 교체할 수 있게 모듈로 분리 | 기획서 §7 확인 대기 항목 |
| A5 | 모든 시드 데이터는 명백한 가상값(홍길동, 테스트교육1 등) + "임의 생성 데이터" 표시 | 기획서 §8 |

**기술 스택 (권장): Next.js (App Router) + SQLite(better-sqlite3 or Prisma) + OpenAI SDK + Upstage API + pdf.js(원본 미리보기)**
- 한 프로세스로 UI·API·DB가 다 돌아가서 배포·시연이 가장 단순함
- 대안: Python(FastAPI) + React — 규칙기반 PDF 파싱 라이브러리(pdfplumber)가 더 강하지만 프로세스가 2개가 됨

### AI 파이프라인 (OCR = Upstage / 추론 = OpenAI)
| 태스크 | 담당 | 근거 |
|---|---|---|
| 신청서(고정폼) 필드 추출 | **Upstage Information Extract** (스키마 고정 JSON, 자체 OCR 내장) | 결정적·비-LLM, 고정양식에 최적 |
| 이수증/결과보고서(폼 제각각) 텍스트화 | **Upstage OCR** (또는 Document Parse) | 한글·저해상도 정확도 높음 |
| 문서 종류 판별 / 이수증 필드 구조화+신뢰도 / 신청↔서류 대조 / 직무부합 근거 | **OpenAI GPT-5.6 Terra** | Terra = "내부 도구·문서 분석"용 티어. 저volume이라 비용 사실상 무시 가능 |
- reasoning effort: 판별·추출은 low~medium, **직무부합 근거·대조 판단은 high**
- **5.6 미제공 시 폴백**: GPT-5.4($2.50/$15) 또는 GPT-5.4-mini($0.75/$4.50)
- 신뢰도 3단계(고/중/저)는 OpenAI 구조화 출력의 필드별 확신도 → 저신뢰는 "검토 필요" 큐로 라우팅

## 1. 데이터 모델 (확장 훅 포함)

```
Employee(직원)          — id, 이름, 부서, 담당업무, 잔여포인트
  └─ Case(신청 건)      — id, employee_id, 교육유형(자율|인사팀주관|일반 ← 확장 훅),
                          교육명, 예상비용, 상태, 반려사유, prev_case_id(재신청 연결),
                          상태전이 타임스탬프들(접수일, 승인일, 서류도착일, 환급일)
       └─ Document      — id, case_id, 종류(신청서|이수증|결과보고서), 파일경로,
                          추출필드 JSON(필드별 값+신뢰도 고/중/저), AI판별 문서종류
       └─ Review        — AI 직무부합 근거문, 세영님 교정 내용(축적 → few-shot 재사용)
```

- 상태: `심사대기 → 이수대기 → 환급대기 → 완료`, 분기 `반려`(사유 필수) → 재신청 시 새 Case + prev_case_id 연결
- **사람(Employee) 중심 저장 구조** — 나중에 "이 직원 총 이수시간" 집계로 자연 확장 (기획서 §5 확장 훅)

## 2. 단계별 계획 (각 단계 = 검증 가능한 목표)

### Phase 1 — 뼈대: 데이터 모델 + 상태 머신
- 프로젝트 스캐폴드, DB 스키마, Case 상태 전이 로직(허용 전이만 가능, 반려 시 사유 필수)
- **검증**: 상태 전이 단위 테스트 통과 (불법 전이 차단, 반려사유 누락 시 거부)

### Phase 2 — 화면 3층 + 가상 데이터
- 층 1 할일 큐(처리대기/검토필요/경과알림, "저신뢰+방치임박" 상단 정렬)
- 층 2 상세 검토 화면(좌: PDF 미리보기, 우: 추출 필드+신뢰도 색상, 상단 주의 배너, 하단 승인/반려)
- 층 3 칸반(5열), 상단 탭 [자율교육(활성)/인사팀주관(준비중)/일반(준비중)]
- 가상 시드 데이터 15~20건(각 상태 골고루 + 저신뢰·불일치·경과 케이스 포함)
- **검증**: 브라우저에서 "신규 접수 → 승인 → 이수증 도착 → 환급 완료" 전체 시나리오 클릭스루
- ※ AI 없이도 도구가 동작하는 상태를 먼저 확보 — 이후 단계가 실패해도 데모 가능

### Phase 3 — PDF 업로드 + AI 추출 파이프라인
- 드롭존: 여러 파일 일괄 투입 → AI가 문서 종류 자동 판별(신청서→새 건 / 이수증→기존 건 매칭)
- 추출: 이름·부서·교육명·금액·수료시간 + 필드별 신뢰도 3단계 → 저신뢰 건은 "검토 필요" 큐로 라우팅
- 가상 양식 PDF 6~8종 제작(신청서 고정폼 + 이수증은 일부러 서로 다른 폼으로)
- 파서 인터페이스 분리: `parseApplication()`은 지금 LLM → 실제 양식 도착 시 규칙기반으로 교체
- **검증**: 가상 PDF 세트 전량 투입 → 문서종류 판별 100%, 필드 추출 오류 건이 저신뢰로 플래그되는지 확인

### Phase 4 — 대조·매칭 로직 (환급 검토)
- 이수증 ↔ 신청 건 자동 매칭(이름+교육명 유사도), 애매하면 후보 제시 후 세영 님 선택
- 신청내용 ↔ 제출서류 대조: 교육명·금액·이름 불일치 하이라이트 + 주의 배너 한 문장 생성
- **검증**: 금액 불일치·교육명 상이·동명이인 시나리오 각각 테스트 → 전부 "검토 필요"에 노출

### Phase 5 — 직무 부합 근거 + 교정 축적
- 심사 단계에서 담당업무 ↔ 교육명 부합 근거문 생성(신뢰도 표기), 잔여 포인트 자동 계산·표시
- 세영 님 교정 저장 → 이후 근거 생성 프롬프트에 few-shot으로 주입 (콜드스타트 해법)
- **검증**: 교정 3건 입력 후 유사 건에서 근거문이 교정 방향을 반영하는지 확인

### Phase 6 — 알림·마감 처리
- 이수 대기 N주 경과 자동 표시(N은 설정값, 기본 4주 — 실제 기한 확인 후 조정)
- 반려 건 재신청 시 이전 반려 사유 표시
- **검증**: 시드 데이터의 경과 건이 "경과 알림" 큐에 뜨는지, 재신청 카드에 이전 사유가 보이는지

### Phase 7 — 데모 마감
- "임의 생성 데이터" 배지 상시 표시, 심사위원 데모 시나리오용 시드 정비
- (선택) KPI 표시: 수기입력 0회·건당 처리시간 등 §6 지표를 화면에 노출
- **검증**: 처음부터 끝까지 데모 리허설 1회 무중단 통과
